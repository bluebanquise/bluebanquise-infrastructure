The project

The need

While building a small HPC center in the FabLab La Fabrique du Loch, at Auray – France, we needed something simple, to manage our cluster and our workstations, but also very flexible as we are working with enterprises and universities (some wants Centos, others Ubuntu, etc).

We also wanted something with less scripts as possible, to easily maintain the product and spend less time managing the cluster to save spare time doing interesting things.

This is how Banquise and then BlueBanquise were born.

What it can do

BlueBanquise is a generic stack, made to handle simple workstations IT room, HPC cluster, enterprise network, etc. We try to keep it as simple as possible, having advanced features optional and hidden.

Supported OS and platform

In its current state, stack can deploy on mixed Linux distributions and platform at the same time, including:
  • Centos/RHEL 7.6 and 8.0, with all features
  • Ubuntu 18.04 with limited features
  • OpenSuse Leap 15.1 with very limited features (PXE able only)
Depending of the demand, Ubuntu or OpenSuse can be fully implemented.
Tested platform were x86_64 for Centos/RHEL/OpenSuse Leap and arm64/x86_64 for Ubuntu.

This ability to handle multiple Linux distributions at the same time allows to easily handle exotic hardware or customer needs.

Stack data

All stack data are stored in plain text files, in YAML format, using Ansible inventory folders structure. No binaries + no databases = fully editable and tunable with minimal knowledge.

Roles

The stack is fully modular, and provides natives core and addons Ansible roles. Roles are nearly all independent from each other's, to be easily replaced by custom ones.

Core

  • /etc/hosts file
  • DHCP (server) with optional advanced features
  • DNS (server/client)
  • Time synchronisation (serber/client)
  • Full advanced PXE stack, based on verbosity and flexibility
    • Tftp based on atftp (with verbosity on by default)
    • Http based on Apache for all repositories and images
    • iPXE custom ROMs with menu, to handle all exotic hardware
      • EFI / Legacy
      • PXE / iPXE native ROM
      • CD or USB boot to PXE when no native PXE available (or stupidly made BIOS/EFI)
  • Repositories (server/client)
  • NIC configuration (based on files or using Ansible NMCLI module)
  • Rsyslog (server/client)
  • Conman (active server)
  • NFS (server/client)

Addons

  • Slurm (basic configuration, master/nodes)
  • Clustershell groups based on Ansible groups
  • Basic OpenLdap (server/client) with phpldapadmin (currently unsecure)
  • Very basic user’s management for very small clusters with a single login node to replace Ldap
  • Prometheus configuration (Prometheus, Alertmanager, NodeExporter, with basic configuration and already few alerts)

Modularity

BlueBanquise has been design to be as modular as possible. As a consequence, data in inventory can be manipulated as desired, and all Ansible roles are tunable or replaceable. See BlueBanquise more like a skeleton, ready to handle flesh.

Specific advanced features

The stack also as few native mechanisms under the hood. These features are a need for us, so we added them.

Multi Icebergs

This feature is sometime called multi islands in HPC or Cloud.

BlueBanquise can split cluster into independent and autonomous groups of equipment. These new subclusters are called icebergs, each iceberg being managed by a group of management nodes and isolated from the others (but can be reached through interconnect if exist and asked).

This feature is mainly used:
  • When there is a need to separated parts of the cluster (our case, to provide dedicated small sub-clusters to some customers).
  • When there is a need to run an old cluster aside a new one keeping both in production.
  • When there is a need to scale to very large number of hosts.

Accelerated mode

Ansible is famous for its simplicity, but also for being slow, even with Mitogen.

Accelerated mode is a POC we designed to heavily accelerate critical templates rendering. It is based on some retro engineering we made on Ansible. It also reduces memory usage.

Accelerated mode is just a basic trick we found in the inventory, that requires no Ansible modifications.

Note that this feature is only at POC state for the time being, as it needs extended developments in roles templates. We are waiting for the stack being more "mature" to fully implement it.
